
这套C端用户域的设计非常成熟，不仅覆盖了用户从注册、信息管理到等级积分的全生命周期，而且在安全性和扩展性上都考虑得相当周全。它已经具备了支撑一个中大型电商系统用户模块的能力。

### ✅ 核心亮点

1.  **清晰的领域划分**
    将用户域拆分为 `usr_p0` (核心基础) 和 `usr_p1` (等级积分) 两个部分，这种模块化设计让结构非常清晰，便于独立维护和扩展。

2.  **稳健的核心表设计 (`usr_users`)**
    *   **灵活的身份体系**：`username`, `email`, `phone` 字段均允许为 `NULL`，这为支持多种登录方式（如第三方OAuth登录）提供了极大的灵活性，用户可以先注册再逐步绑定信息。
    *   **完善的安全实践**：
        *   使用 `password_hash` 存储密码哈希，符合安全规范。
        *   `email_verified` 和 `phone_verified` 字段用于标记验证状态，是保障账户安全的基础。
        *   `usr_login_histories` 表详细记录了登录IP、设备、地点和结果，为安全审计和异常行为追踪提供了数据支持。
    *   **标准的审计字段**：`created_at`, `updated_at`, `deleted_at` 是标配，特别是 `deleted_at` 实现了逻辑删除，保证了数据的可追溯性。

3.  **规范的扩展设计 (`usr_infos`)**
    将用户的扩展信息（如性别、生日、地址偏好等）从主表中剥离，放入 `usr_infos` 表，通过 `user_id` 进行一对一关联。这是一种非常经典的数据库优化手段，可以有效提升核心 `usr_users` 表的查询性能。

4.  **灵活且强大的等级积分体系**
    这是整个设计中最出彩的部分，它没有将规则写死在代码里，而是通过数据表进行配置，实现了极高的灵活性。
    *   **等级定义 (`usr_levels`)**：
        *   通过 `min_points` 和 `max_points` 定义等级区间。
        *   使用 `discount_rate` (千分比) 和 `points_multiplier` (倍数) 来量化会员权益，设计精确。
        *   预留 `benefits` JSON 字段，可以方便地扩展如生日礼包、专属优惠券等非结构化权益，无需频繁修改表结构。
    *   **积分流水 (`usr_points`)**：
        *   `points` 字段用正负数表示增减，设计简洁。
        *   `source` 和 `source_id` 的组合可以清晰地追踪每一笔积分的来源（如订单、评价、签到），便于对账。
        *   `expire_at` 和 `status` 字段为处理积分过期、待确认等复杂状态提供了支持。
    *   **规则配置化 (`usr_level_rules`, `usr_points_rules`)**：
        *   将升级/降级规则和积分获取规则抽离成独立的配置表，运营人员可以通过后台动态调整规则，而无需研发介入，极大地提升了业务响应速度。

### 🤔 可优化的细节

尽管整体设计非常出色，但在一些细节上仍有打磨空间：

1.  **`usr_levels` 表的字段冗余**
    *   **问题**：同时存在 `level` (等级数值) 和 `min_points`/`max_points` (积分区间) 字段。当等级规则变更时（例如调整黄金会员的积分门槛），需要同时维护这两个字段，容易出现数据不一致的风险。
    *   **建议**：可以考虑只保留 `level` 和 `min_points`。用户的当前等级可以通过其累计积分 (`usr_infos.total_points`) 与 `usr_levels.min_points` 进行比较动态得出。这样可以确保等级计算的唯一数据源，避免冗余。

2.  **`usr_points_rules` 表的 `rule_value` 类型**
    *   **问题**：`rule_value` 字段使用 `varchar` 类型来存储所有规则的值。虽然灵活性高，但会带来一些问题：
        *   **类型安全**：无法在数据库层面保证值的类型（例如，`earn_rate` 应该是数字，但可能被误存为字符串）。
        *   **查询效率**：如果需要基于某个数值规则进行查询或排序，对 `varchar` 字段进行操作效率较低。
    *   **建议**：可以考虑将 `rule_value` 拆分为不同类型字段，如 `value_int`, `value_decimal`, `value_string`，或者统一使用 `JSON` 类型来存储，并在应用层进行严格的类型校验。

3.  **默认地址的唯一性约束**
    *   **问题**：`usr_addresses` 表中用 `is_default` 字段标记默认地址。在应用层逻辑中，需要确保一个用户只有一个默认地址，但这在数据库层面没有强制约束，极端情况下可能出现数据不一致。
    *   **建议**：可以在 `(user_id, is_default)` 上创建一个唯一索引，但需要将 `is_default` 的值从 `0/1` 改为 `NULL/1` (即非默认地址的 `is_default` 字段存为 `NULL`)。因为在大多数数据库中，唯一索引允许有多个 `NULL` 值，但只允许一个特定值（如 `1`）。这是一个更高级但更严谨的做法。

### 总结

总的来说，这套C端用户域的表设计是一个**非常优秀的范本**。它结构清晰、考虑周全，特别是在等级积分体系的配置化设计上展现了很好的前瞻性。对于绝大多数电商项目而言，这套设计已经绰绰有余。上述的优化建议属于锦上添花，可以在项目发展到特定阶段，遇到相应痛点时再进行迭代。

### 已实施的优化改造

针对上述可优化点，已完成以下改造：

1.  **`usr_levels` — 移除 `max_points` 字段**
    *   `sql/usr_p1.sql`: 删除了 `max_points` 列，等级仅通过 `min_points` 门槛定义
    *   所有 seed 数据同步移除 `max_points` 引用
    *   用户等级通过累计积分 `usr_infos.total_points` 与 `usr_levels.min_points` 动态比较得出

2.  **`usr_points_rules` — `rule_value` 拆分为类型化字段**
    *   `sql/usr_p1.sql`: `value_int`、`value_decimal`、`value_string` 三列替代 `rule_value varchar(255)`
    *   当前规则数据使用 `value_int` 存储，后续可按需使用其他类型

3.  **`usr_addresses` — 默认地址唯一约束**
    *   `sql/usr_p0.sql`: `is_default` 改为 `DEFAULT NULL`，新增唯一索引 `uk_user_default(user_id, is_default)`
    *   利用 MySQL 唯一索引允许 NULL 的特性，数据库层面保证每用户最多一个默认地址
    *   非默认地址 `is_default = NULL`，默认地址 `is_default = 1`
